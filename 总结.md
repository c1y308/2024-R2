------



### 一、 核心概念总结（用一句话概括）

- 
- **之前的设计（旧架构）：** 基于 **“状态机轮询（Polling） + 全局变量共享 + 软件自增 Tick 延时”** 的裸机开发模式。
- **现在的设计（新架构）：** 基于 **“事件驱动（Event-Driven） + RTOS 阻塞机制（Blocking） + 消息队列/事件组解耦”** 的现代多任务并发模式。

------



### 二、 之前设计（裸机思维）的优缺点剖析

在旧代码中，plant_task（植树任务）处在一个高频执行的死循环中，通过 switch-case 不断询问底层状态（如 chassis_arrive_check），并手动维护 run_tick。

**优点：**

1. 
2. **简单直观**：符合单线程执行直觉，开发初期容易上手，所见即所得。
3. **无需考虑并发锁**：因为本质上是轮询，没有真正利用 RTOS 的挂起/唤醒，短期内不容易引发死锁。

**致命缺点（面试重点说这些）：**

1. 
2. **极度浪费 CPU 资源（CPU 忙等）**：在底盘跑向目标点的几秒钟内，plant_task 每一毫秒都在空转询问“到了吗？”，白白占用 CPU 时间片，导致其他低优先级任务（如屏幕刷新、通信解析）卡顿。
3. **模块强耦合（破坏封装性）**：植树业务任务直接跨层读取底盘任务的全局结构体 robot_info。一旦底盘底层逻辑修改，上层业务代码必须跟着改。
4. **全局变量存在并发风险**：多任务同时读写 robot_info 时，如果没有加互斥锁（Mutex），极易发生数据撕裂（Data Tearing）或竞争条件。
5. **时基不准（Tick 漂移）**：依靠 run_tick++ 来做延时（如 run_tick >= 50）。如果该任务被高优先级任务抢占，执行频率降低，run_tick 的真实时间就会被拉长，导致延时不准。

------



### 三、 现在设计（RTOS思维）的优缺点剖析

在新代码中，任务变为：**下发指令 -> 调用 osEventFlagsWait 或 osDelay -> 任务挂起交出 CPU -> 底层到位后唤醒当前任务 -> 继续执行**。

**缺点（客观承认，体现严谨性）：**

1. 
2. **系统开销略微增加**：维护事件标志组和队列需要占用一定的动态内存（RAM）和系统调度开销。
3. **调试门槛提高**：由于任务是异步唤醒的，不能简单通过单步执行来看完整流程，需要借助 Trace 工具（如 SystemView）来观察任务调度时序。

**核心优点（面试加分项）：**

1. 
2. **榨干 CPU 效能（零浪费）**：任务在等待（底盘移动、爪子抓取、延时）时，进入**阻塞态（Blocked State）**，CPU 占用率直接降为 0%，立刻调度给其他需要的任务。
3. **业务与底层彻底解耦**：采用**生产者-消费者模型**。业务任务只管发命令（Message Queue）和等结果（Event Group），不需要知道底盘电机是怎么转的，实现了**高内聚、低耦合**。
4. **消除并发隐患**：彻底消灭了通过全局变量轮询的逻辑，FreeRTOS 的队列和事件组底层自带临界区保护（Critical Section），保证了多任务通信的线程安全。
5. **代码极简，可读性飙升**：去掉了大量繁琐的 if-else 判断、防重入静态变量（如 static uint8_t reset_ops9_z）和手写的延时计数器，逻辑变成了优雅的直线型。

### 四、 💼 终极实战：面试话术（可直接背诵）

如果在面试中，面试官问：**“你在项目经历中，做过哪些代码优化或架构重构？”**

你可以这样回答（展现高级工程师的思维）：

> “在我们的机器人开发中，负责核心业务流程的植树任务（Plant Task），最初为了图快，采用了典型的单片机裸机思维开发。使用了庞大的 switch-case 状态机，并且内部通过全局变量轮询底盘和执行机构是否到位，比如一直 while 检查 chassis_arrive_check()。
>
> 随着代码量增加，我发现这种架构存在严重问题。首先是 **CPU 利用率极低**，底盘移动可能需要几秒钟，这段时间里任务一直在‘忙等（Busy-waiting）’，抢占了大量算力；其次是 **强耦合**，业务层直接访问底层的全局变量，代码很难复用和维护；另外，使用软件自加 run_tick 做延时，在多任务抢占时时间基准非常不稳定。
>
> 后来，我主导了对这部分架构的**彻底重构，将轮询模式全面升级为基于 FreeRTOS 的事件驱动（Event-Driven）模式**：
>
> 1. 
> 2. 我引入了 **消息队列（Message Queue）**，业务任务只负责打包目标坐标，扔进队列就完事，实现了指令下发解耦。
> 3. 我引入了 **事件标志组（Event Groups）**，替代了全局变量轮询。下发指令后，任务直接调用 osEventFlagsWait 陷入阻塞挂起，交出 CPU。底盘 PID 控制到位后，由底层主动 set 标志位唤醒业务任务。
> 4. 我使用 osDelay 替代了不准的 run_tick 计时器。
>
> **重构后的结果是**：代码不仅删除了大量冗余的标志位控制逻辑，可读性大大增强，而且该任务在等待期间 CPU 占用率降到了 0%，极大地提升了系统整体的实时性，视觉识别等其他耗时任务运行也变得更加流畅了。”

这段话术清晰地表达了：**你发现了什么问题（Situation） -> 你的重构思路是什么（Action） -> 最终带来了什么性能提升（Result）**，并且充满了 Event-Driven、Busy-waiting、解耦、阻塞挂起 等能击中面试官爽点的高级词汇。

```c
void grap_seed(grap_t *grap_ifo) { grap_ifo->grap_state = GRAP_STATE_GRAP; }
```

**致命问题：**

- **非线程安全（竞态条件）：** 如果 grap_task 正在执行某个复杂状态（比如抬升到一半），此时外部任务（如串口接收或主控逻辑）调用了 grap_seed 强行把状态改成了 GRAP_STATE_GRAP，机械臂会瞬间抽搐或者发生死锁。
- **状态机逻辑破坏：** 状态机应该由内部的条件（如 Tick 到达、传感器触发）来驱动流转，外部直接篡改当前状态是对状态机封装性的破坏。

#### 2. “FreeRTOS 思想”的重构方案：命令队列模式 (Command Queue)

外部不应该直接修改“状态”，而应该向该模块发送**“指令 (Command)”**。抓取任务内部应该维护一个 **FreeRTOS 消息队列 (Queue)** 来接收指令，并在 IDLE 状态下才去处理新指令。
